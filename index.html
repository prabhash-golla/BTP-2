The Evolution of Access Control

1

Traditional PKE & PKI

Public Key Encryption relies on certificates (CAs) to bind an identity to a public key.

  • Challenge: Costly infrastructure and point-to-point communication requirements.
  • Challenge: Revocation management.
2

Identity-Based (IBE)

Shamir (1984). Uses a user's unique identity directly as the public key.

  • Advantage: No certificates needed.
  • Flaw: Introduces the Private Key Generator (PKG) problem.
3

Attribute-Based (ABE)

Sahai & Waters (2005). Embeds boolean access policies directly into the math.

  • Advantage: One-to-many encryption (e.g., Dir OR Mgr).
  • Flaw: Retains the PKG central authority vulnerability.

The Key Escrow Bottleneck

The primary barrier to widespread ABE deployment in zero-trust cloud environments is the fundamental trust model required during key generation.

The Central PKG Vulnerability

To issue a functional secret key, the PKG must use the Master Secret Key (MSK):

$SK_{attr} \leftarrow f(MSK, Attributes)$
  • Single Point of Failure: The PKG can decrypt every ciphertext in the network.
  • Malicious Provider: If the cloud provider is compromised, system-wide privacy can be permanently lost.

The Reg-ABE Paradigm Shift

Registered ABE replaces the PKG with a non-privileged, deterministic Key Curator.

$(sk_u, pk_u) \leftarrow KeyGen(crs_{local})$
  • Client-Side Keygen: Users compute their own secrets.
  • Trustless Aggregation: Curator only manages public keys.
  • Post-Compromise Security: Curator database leaks reveal zero user secrets.

BTP-I Recap: IBE from Standard CDH

Prior to this semester, we implemented the Döttling-Garg (CRYPTO 2017) non-black-box architecture to prove escrow-free IBE is theoretically possible using only standard Computational Diffie-Hellman (CDH) assumptions.

Chameleon Encryption & Garbled Circuits

Instead of compiling field mathematics directly into gates, we used decryption-as-path-verification.

  • Chameleon Hash: A collision-resistant hash with a trapdoor.
  • Circuit Chain: Ciphertext contains $n+1$ garbled circuits $(\tilde{P}_0, ..., \tilde{P}_{n-1}, \tilde{T})$.
  • Logic: Output of $\tilde{P}_i$ evaluates encryptions of labels for $\tilde{P}_{i+1}$.

Key Insight: Each circuit unlocks the next in a cryptographic chain reaction.

BTP-I Post-Presentation Improvements

Following the BTP-I presentation, we introduced optimizations to mitigate the massive computational bottlenecks of the garbled chain.

1. Tree Arity Optimization

In the naive construction, a 512-bit identity requires a depth of 512 sequential circuits.

  • We restructured the evaluation tree to process the identity in blocks ($128$-bit segments).
  • $4 \times 128$ bits for a 512-bit ID significantly reduces circuit depth.
$d = n / \log_2(a)$

2. Label Chunking & AES Integration

We utilized the active label $\tilde{\Pi}_i$ to derive a symmetric key $K$, using standard AES encryption to secure the path.

$K \leftarrow KDF(\tilde{\Pi}_i)$
$C_{out} = AES_K(Next\ Label)$
$ct = C_{out} \oplus GC\ Payload$

Reduced wire labels from 512-bit to 128-bit.

The Pivot to Pairings

Bilinear pairings perform equivalent cryptographic checks with native group operations — no circuit compilation needed.

Algebraic Foundations & Assumptions

Transitioning to pairing-based cryptography requires understanding the specific hardness assumptions that anchor semantic security.

Subgroup Decision Assumption

Anchors legacy composite-order cryptography.

  • Group order $N = p_1 p_2 p_3$
  • Adversary cannot discern if an element resides in the full group $\mathbb{G}_N$ or a constrained subgroup $\mathbb{G}_{p1}$.
  • Allows independent subgroup cancellation, but requires very large (1024-2048 bit) key sizes.

$k$-Linear ($k$-Lin) Assumption

Anchors modern prime-order cryptography. Generalizes DDH ($k=1$) and DLIN ($k=2$).

  • Given vectors in an affine subspace of dimension $k$.
  • Computationally intractable to determine if a challenge vector belongs to that specific subspace or is uniform.
  • Allows for 256-bit parameter sizes.

Asymmetric Pairings

A pairing is a computable map $e: \mathbb{G}_1 \times \mathbb{G}_2 \rightarrow \mathbb{G}_T$ bridging elements from two prime-order groups. The choice of pairing type profoundly affects efficiency and security.

Type II Pairing

An efficiently computable homomorphism exists in one direction:

$\phi: \mathbb{G}_2 \rightarrow \mathbb{G}_1$
  • Structure: $\mathbb{G}_1 \neq \mathbb{G}_2$, but map $\phi$ links them.
  • Advantage: Some protocols simplify with directional maps.
  • Drawback: The homomorphism leaks structural info, limiting proof strategies.

Type III Pairing

No efficient homomorphism exists between $\mathbb{G}_1$ and $\mathbb{G}_2$ in either direction.

$e: \mathbb{G}_1 \times \mathbb{G}_2 \rightarrow \mathbb{G}_T$
  • Bilinearity: $e(u^a, v^b) = e(u,v)^{ab}$
  • Non-degeneracy: $e(g_1, g_2) \neq 1_{\mathbb{G}_T}$
  • Computability: Poly-time for all valid inputs.

Why Type III?

The absence of any connecting homomorphism provides the strongest security guarantees:

  • Compact Keys: Built on BLS12-381 curves with ~256-bit parameters.
  • Faster Compute: Smallest group elements yield fastest pairings.
  • Tight Proofs: No structural leakage enables tighter security reductions under $k$-Lin.
  • Industry Standard: Used in zk-SNARKs, BLS signatures, and Ethereum 2.0.
$e(g^a, g^b) = e(g,g)^{ab}$ — enables hidden multiplicative checks without revealing exponents

Dual Pairing Vector Spaces (DPVS)

To simulate composite subgroup cancellations in highly efficient prime-order spaces, we shift dimensions: replacing numbers with vectors.

Orthogonal Bases

Setup algorithm generates two strictly orthogonal bases:

$\mathbb{B} = (b_1, b_2, b_3, b_4) \in \mathbb{G}_1^4$

$\mathbb{B}^* = (b_1^*, b_2^*, b_3^*, b_4^*) \in \mathbb{G}_2^4$

Dot-product pairing yields orthogonality:

$e(b_i, b_j^*) = \begin{cases} g_T^\psi & \text{if } i = j \\ 1 & \text{if } i \neq j \end{cases}$

Architectural Roles

  • Master Public Key & Ciphertexts live in the span of $\mathbb{B}$ (in $\mathbb{G}_1$).
  • Secret Keys & Curator's HSK live in the span of $\mathbb{B}^*$ (in $\mathbb{G}_2$).
  • Blinding Subspaces: Cross-terms generated during curator aggregation are placed in specific subspaces (e.g., spanned by $b_2, b_3$).

Result: During decryption, orthogonality mathematically guarantees $e(b_{data}, b_{blind}^*) = 1$, canceling the blinding factors while preserving the secret payload.

Linear Secret Sharing Schemes (LSSS)

Raw boolean logic must be mathematically converted into matrices to enforce policies within pairing environments.

LSSS Construction

  • Policy Matrix: $\ell \times n$ matrix $M$ with mapping function $\rho$ (rows $\to$ attributes).
  • Secret Vector: Encryptor hides secret $s$ inside $\vec{v} = (s, r_2, ..., r_n)^\top$.
  • Share Generation: Multiplication $M \cdot \vec{v}$ yields the shares.
  • Reconstruction: If attributes satisfy the structure, user derives constants $c_i$ to solve $\sum c_i(M_i \cdot \vec{v}) = s$.
Non-Monotone Span Programs: LSSS mathematically represents NOT-gates via orthogonal vectors, natively supporting exclusionary logic (e.g., Director AND NOT Contractor).

Matrix Projection Visualization

$$ \begin{bmatrix} 1 & 0 & -1 \\ 0 & 1 & 0 \\ 1 & 1 & 1 \end{bmatrix} \times \begin{bmatrix} s \\ r_2 \\ r_3 \end{bmatrix} = \begin{bmatrix} \lambda_1 \\ \lambda_2 \\ \lambda_3 \end{bmatrix} $$

*Shares $\lambda_i$ embed the payload. Only correct linear combinations of $\lambda_i$ cancel out random factors $r_x$.

The Key Curator Model: Core Algorithms

The centralized PKG is replaced by a transparent curator executing deterministic polynomial-time algorithms.

1

Setup($1^\lambda, 1^N$)

Generates CRS including DPVS bases. Setup party goes offline permanently.

2

KeyGen($crs$)

Run locally. User samples $sk_i \leftarrow \mathbb{Z}_p$, publishes $pk_i = g^{sk_i}$. Curator never sees $sk_i$.

3

Register

User sends verified attributes and $pk_i$ to curator for the public directory.

4

Update

Curator aggregates keys to form global $mpk$ and computes $hsk_i$ for cross-term blinding.

5

Encrypt

Uses global $mpk$ and policy matrix $y$. Ciphertext size is $O(|y|)$.

6

Decrypt

Combines local $sk_i$ and public $hsk_i$. Bilinear pairing isolates exact slot. LSSS unblinds message.

Setup → KeyGen → Register → Update → Encrypt / Decrypt

The Curator never possesses any secret key material — it performs only public-key aggregation, a deterministic, auditable operation. Even if compromised, no user's secret key is revealed.

Algebraic Aggregation & Cross-Terms

Aggregation ($Update$)

User $i$ samples local secret $r_i \leftarrow \mathbb{Z}_p$, publishes $T_i = g^{r_i}$. The curator aggregates purely via multiplication:

$\hat{T} = \prod T_j = g^{\sum r_j}$

Because MPK incorporates exponents from all users, a single user requires the HSK to mathematically filter out the "cross-terms" belonging to everyone else.

Slot Isolation ($Decrypt$)

Curator packages blinding factors $V_{j,i} = A_j^{r_i}$ into $hsk_i$. User $i$ leverages pairing to check and isolate their slot:

$e(C, h_{N+1-i}) = e(\Lambda_i, g_2) \cdot e(h_i^{r_i}, h_{N+1-i})$

Once cross-terms are cleanly canceled out, the user proceeds to the Predicate Encoding ($LSSS$) check.

Predicate Encodings ($LSSS$ Policy Checks)

Maps boolean access formulas into matrices over a finite field $\mathbb{Z}_p$. Let $x \in \mathcal{X}$ (attributes) and $y \in \mathcal{Y}$ (policy).

1. Encoding Components

Ciphertext Encoding ($C_x$): Mathematical attribute vector.
Key Encoding ($K_y$): The LSSS access matrix.
Decryption Vector ($d_{x,y}$): Computable only if $P(x,y)=1$.

2. The Reconstruction

User computes $M_{x,y} d_{x,y}^\top = e_1^\top$, effectively wiping the blinding factors. If $P(x,y)=0$, shared secret $\alpha$ remains perfectly statistically hidden.

Composite Matrix

$$ M_{x,y} = \begin{pmatrix} \alpha_y & 0 \\ K_y & C_x \end{pmatrix} $$
Secret Payload $\alpha_y$
Blinding Subspace $K_y, C_x$

Decryption Flow

Attributes → $C_x$ + Policy → $K_y$ → Build $M_{x,y}$ → Solve $d_{x,y}$ → Recover $\alpha$

Vector Commitments & Merkle Registration

Achieving $O(\log N)$ registration efficiency and trustless auditing.

The Registration Tree

Public keys are compressed into a digest via Vector Commitments in pairing group $\mathbb{G}$:

$C = \prod_{i \in [N]} h_i^{pk_i}$
  • Commitments form leaves of a Merkle Tree.
  • The Root Hash definitively acts as the MPK.
  • Any user can request $O(\log N)$ sibling hashes to verify inclusion without trusting the Curator.

Slotted Merging

Direct dynamic joining is complex. We implement a "Powers-of-Two" mechanism over slotted capacities.

Efficiency Optimization via Prime-Order Groups

Moving from composite subgroups to prime-order vector subspaces (via DPVS) drastically reduces overhead.

Metric Composite-Order Group Prime-Order Vector Space
Math Basis $N = p_1 \cdot p_2 \cdot p_3$ (Factoring Hard) Single Prime $p$ (e.g. BLS12-381)
Isolation Tool Subgroup $\mathbb{G}_{p1}$ cancels $\mathbb{G}_{p2}$ Orthogonal vector subspaces $\mathbb{V}_1 \perp \mathbb{V}_2$
Security Proof Subgroup Decision Assumption Decision Linear ($k$-Lin) Assumption
Ciphertext Size Large (~1024-bit per element) Small (~256-bit per element)
Master Public Key $O(L^2) \rightarrow$ $\approx 15 \text{ MB}$ $O(L) \rightarrow$ $\approx 7.5 \text{ KB}$ (>2000x smaller)

2000x Smaller MPK

From ~15 MB down to ~7.5 KB, enabling mobile and IoT deployment scenarios previously impossible.

4x Faster Pairings

Prime-order pairings on BLS12-381 are ~4x faster than composite-order equivalents at the same security level.

Standard k-Lin Security

Security reductions under $k$-Lin are tighter and better understood than composite subgroup decision assumptions.

Kronecker Products & QA-NIZK Proofs

QA-NIZK Injection Defense

Quasi-Adaptive Non-Interactive Zero-Knowledge proofs appended to registered public keys prevent adversaries from inserting malformed components.

  • $Verify(crs, pk_i, \pi_i) == 1$ required before curator aggregation.
  • Defends against adaptive chosen-key attacks on the MPK structure.

Kronecker Encodings

LSSS access matrices are mapped onto DPVS basis networks via Kronecker Products ($\otimes$):

$A \otimes B = \begin{pmatrix} a_{11}B & \dots & a_{1m}B \\ \vdots & \ddots & \vdots \\ a_{n1}B & \dots & a_{nm}B \end{pmatrix}$

Full CCA-Security

By combining QA-NIZK proofs of ciphertext well-formedness with one-time signatures binding the ciphertext to the policy, the system is upgraded from CPA (passive) to Chosen-Ciphertext Attack (CCA) resilience, securing the payload against aggressive decryption oracle manipulation.

Simulation-Based Security Proofs

Proving security in Reg-ABE relies on the Dual-System Argument, which controls entropy leakage across sequential key queries.

IND-Reg-ABE Game

  1. Adversary chooses messages $m_0, m_1$ and challenge policy $y^*$.
  2. Challenger encrypts $m_b$ (random $b$) under $y^*$.
  3. Adversary queries $Register(x_i)$ for any attribute set where $P(x_i, y^*) = 0$ (ensuring the challenge policy cannot be satisfied).
  4. Adversary guesses $b'$. Advantage must be negligible: $|\Pr[b'=b] - 1/2| \le \epsilon$.

Hybrid Transition (k-Lin)

  • Game 0: Real IND-Reg-ABE game.
  • Game 1: Replace ciphertext randomness with DPVS blinding.
  • Game 2: Apply k-Lin reduction.
  • Game 3: Dual-system transition (keys become semi-functional).
  • Game 4: Perfect information-theoretic hiding of message payload.

Registered Functional Encryption (Reg-FE)

Standard ABE decryption is strictly binary (all or nothing). Functional Encryption allows a user to evaluate a function $f(x)$ over an encrypted payload $x$, outputting exactly the computed result and revealing nothing else about the data.

Registered Inner-Product FE (Reg-IPFE)

User registers a linear function vector $\vec{y}$. Provider encrypts sensitive data vector $\vec{x}$ (e.g., financial metrics).

$f_{\vec{y}}(\vec{x}) = \langle \vec{x}, \vec{y} \rangle = \sum_{i=1}^n x_i y_i$
  • Ciphertext Size: Weakly compact $O(n \cdot \log L)$ group elements.
  • Security: Adaptive IND-security anchored by $k$-Linear assumption in standard model.
  • Application: Decentralized statistical processing without central trust.

Registered Quadratic FE (Reg-QFE)

Quadratic Evaluation

Allows evaluation of quadratic polynomials (e.g. variances, spatial distance) over blinded arrays.

User registers matrix $F \in \mathbb{Z}_p^{n \times n}$. The ciphertext securely encodes $(\vec{x}_1, \vec{x}_2)$. The decryption sequence computes:

$f(\vec{x}_1, \vec{x}_2) = (\vec{x}_1 \otimes \vec{x}_2) F^\top$

Pre-Constrained Innovation (PReg-QFE)

The Problem: Computing quadratic functions directly produces non-linear terms that break the Curator's clean linear aggregation symmetry.

The Solution: Separate function definition from user registration. The master matrix defining the function space is embedded directly into the CRS during trusted setup.

Curator workload remains purely linear (simple multiplications). Non-linear complexity is enforced dynamically during decryption pairing.

Open Problems & Roadblocks

As this architecture advances, several critical challenges remain at the frontier of decentralized cryptography.

Post-Quantum Reg-ABE

Bilinear pairings on elliptic curves are fundamentally vulnerable to Shor's algorithm. Constructing efficient Reg-ABE from lattice-based assumptions (LWE) without reverting to Garbled Circuits requires novel research into lattice vector commitments.

Compactness in Reg-QFE

Current Reg-QFE suffers from a massive CRS of size $O(n^2 \cdot L^2 \cdot \log L)$. Reducing this to a linear dependency on $n$ and $L$ requires new dual-system algebra techniques.

Multi-Authority Extension

Current Reg-ABE assumes a single Curator. Extending to decentralized multi-authority settings — where multiple independent curators manage disjoint attribute universes — requires cross-domain consistency protocols.

Revocation Efficiency

Revoking a registered user requires Curator to recompute the aggregated MPK and issue new $hsk$ values for all remaining users. Achieving $O(\log N)$ revocation (vs. current $O(N)$) remains an open problem.

Scaling & Accountability Vectors

Traitor Tracing

In Reg-ABE, if a malicious user leaks their locally generated secret key ($sk_i$) or the curator's $hsk_i$ to a third party, it is impossible to identify the source. Integrating mathematical tracking to fingerprint specific registered slots is crucial for enterprise accountability.

Unbounded Scaling without iO

Current pairing-based implementations enforce a fixed maximum capacity $L$ (slots). Removing this limit currently relies on Indistinguishability Obfuscation (iO), which is theoretically strong but has limited practical applicability. Unbounded scaling using standard black-box techniques remains an open challenge.
